\section{CI/CD とは}
\textbf{CI/CD}（継続的インテグレーション／継続的デリバリー）とは、ソースコードの変更を
\textbf{自動でビルド・テスト・デプロイ}する仕組みである。本課題では \texttt{git push} を契機に
\textbf{GitHub Actions} が自動起動し、静的サイトを \textbf{Azure Static Web Apps} へ公開する。
手作業の FTP アップロードが不要になり、ヒューマンエラーを防げる点が利点である。

\begin{center}
\small
\texttt{[ローカルPC]} $\xrightarrow{\text{git push}}$ \texttt{[GitHub(main)]}
$\xrightarrow{\text{Actions}}$ \texttt{[Azure Static Web Apps]}
$\xrightarrow{\text{HTTPS}}$ \texttt{[利用者]}
\end{center}

\noindent\textbf{使用技術:}\ GitHub / GitHub Actions / \texttt{Azure/static-web-apps-deploy@v1} /
Azure Static Web Apps（無料プラン）／静的 HTML + CSS

\section{考察 --- 運用で気づいた課題}
運用中、\textbf{push のたびにワークフローが1回失敗}していた。原因はデプロイ用ワークフローが
\texttt{azure-static-web-apps.yaml}（手動作成）と自動生成版の\textbf{2本}存在し、同じ Static Web App へ
同時デプロイして衝突し、片方が \texttt{Upload Timed Out} でタイムアウトするためである。
不要な旧ファイルを削除して\textbf{デプロイ経路を1本化}すれば安定する。CI/CD では
「1つの成果物へのデプロイ経路は1本に保つ」ことが重要だと学んだ。
