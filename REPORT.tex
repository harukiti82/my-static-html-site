\section{はじめに}
本レポートは，講義で学んだ CI/CD（Continuous Integration / Continuous Delivery）を実際に用いて，
Web サイトをクラウド上へ自動で公開する仕組みを構築し，その過程と結果を記録したものである．
題材として，GitHub Actions と Azure Static Web Apps による\textbf{静的サイトの継続的デプロイ}を中心に扱い，
さらに追加課題として Azure App Service 上に\textbf{動的な Web アプリ}を構築した内容を報告する．

\section{CI/CD の仕組みとシステム構成}
CI/CD とは，ソースコードの変更をきっかけに，ビルドから公開までの一連の作業を自動で行う仕組みである．
本課題では，GitHub のリポジトリへ変更を反映（push）すると，GitHub Actions が自動的に起動し，
Azure Static Web Apps へファイルを配置して公開する．全体の流れは次の通りである．

\begin{center}
\small
\texttt{[ローカルPC]} $\longrightarrow$ \texttt{[GitHub(main)]}
$\longrightarrow$ \texttt{[GitHub Actions]} $\longrightarrow$ \texttt{[Azure Static Web Apps]}
$\longrightarrow$ \texttt{[利用者]}
\end{center}

\noindent
この構成では，開発者は変更を push するだけでよく，サーバーへ手作業でファイルを送る必要がない．
リポジトリには公開する \texttt{index.html}・\texttt{style.css} と，動作を定義する
\texttt{.github/workflows/} 内の YAML を配置している．

\section{デプロイの自動実行}
\texttt{main} ブランチへ push するたびに，GitHub の Actions タブでビルドとデプロイが自動実行される
（付録 図\ref{fig:deploy}）．処理は 1 分ほどで完了し，成功するとログの末尾に公開先の URL が表示される．
Pull Request を作成した場合には，マージ前に内容を確認するためのプレビュー環境も自動で用意される．
実行結果は履歴として残るため，いつ・どのデプロイが成功／失敗したかを後から追跡できる点も利点である．

\section{変更内容の管理}
ソースコードの変更は Git のコミットとして記録され，「いつ・どこを・どのように変えたか」を差分として
確認できる（付録 図\ref{fig:change}）．たとえば見出しの文字色を変更するコミットでは，該当する 1 行のみが
変更点として色付きで示される．この変更を push すると，前節の仕組みによって自動で本番サイトへ反映される．
少量の修正でも，人手の配置を挟まずに公開まで届くことが CI/CD の効果である．

\section{公開サイトと動作確認}
継続的デプロイによって公開されている静的サイトの URL は付録に示す通りで，2026 年 7 月 12 日 時点で
正常に表示される（付録 図\ref{fig:static}）．push した変更が実際にこの URL へ反映されることを確認した．

\section{動的 Web アプリの構築（追加課題）}
追加課題として，Azure App Service 上に動的な Web アプリを構築した．静的サイトが同じ HTML を返すだけで
あるのに対し，こちらは Node.js のフレームワークである Express を用い，\textbf{リクエストのたびに
サーバー側で内容を生成}する（付録 図\ref{fig:dynamic}）．具体的には，アクセス時点のサーバー時刻，
メモリ上で加算されるアクセス回数，フォームに入力した名前を受け取って応答を返す機能を実装した．
App Service は PaaS であり，サーバーの構築や保守を意識せずにアプリを公開できる．なお無料プランのため，
初回アクセス時はコールドスタートにより表示までに時間がかかる．

\section{考察とまとめ}
本課題を通じて，push を起点にビルドから公開までが自動化される CI/CD の流れを，実際の構築を通して
理解できた．手作業での配置が不要になることで，ヒューマンエラーを避けつつ短時間で更新を反映できる．
一方で運用中，デプロイ用ワークフローが 2 本存在して同じサービスへ同時にデプロイし，一方が
タイムアウトで失敗していた．デプロイ経路は 1 本に保つ必要があるという運用上の注意点も学べた．
加えて App Service により，静的配信にとどまらない動的なアプリの公開まで体験できた．

\section*{付録}
\subsection*{A.\ 公開 URL}
構築したサービスの URL は以下の通りである（いずれも 2026 年 7 月 12 日 時点で稼働確認済み）．
\begin{quote}\ttfamily\small
静的サイト：https://orange-stone-0cfe86d00.7.azurestaticapps.net\\
動的アプリ：https://numata-25g1104-express-1833.azurewebsites.net
\end{quote}
※ 静的サイトのドメインには \texttt{.7.} が含まれる．これを省いた URL は 404 になるため注意する．
