\section{はじめに}
本レポートでは，アプリケーションの更新時に自動でデプロイ作業が行われる仕組みである
CI/CD（Continuous Integration と Continuous Delivery）を取り上げ，手動でのデプロイと自動デプロイを
比較する．あわせて，今回構築した 2 つのサービス，すなわち GitHub Actions による自動デプロイで
公開した\textbf{静的 Web サイト}（Azure Static Web Apps）と，サーバー側で処理を行う
\textbf{動的 Web アプリ}（Azure App Service）を題材に，CI/CD と PaaS の利点をまとめる．

\section{CI/CD 導入前（手動でのデプロイ）}
CI/CD が用いられていない場合，開発したファイルを開発者自身がサーバーへ配置して，はじめて
ユーザーが利用できる状態になる．変更のたびに人手で同じ配置作業を繰り返す必要があり，次の課題がある．
\begin{itemize}
  \item \textbf{ヒューマンエラーの発生}：手作業で配置するため，操作ミスや手順の抜けが起こりやすい．
  \item \textbf{手間と時間がかかる}：修正のたびにローカルから同じ内容を送り直す必要があり，
        更新回数が増えるほど負担が大きくなる．
\end{itemize}

\section{CI/CD 導入後（自動デプロイ）}
CI/CD を導入すると，コードの変更をリポジトリへ反映するだけで，ビルドから公開までが自動で実行される．
本課題では，ローカルで編集した HTML / CSS を GitHub の \texttt{main} ブランチへ push するだけで，
GitHub Actions が起動し，Azure Static Web Apps へファイルが自動でアップロードされる仕組みを構築した．
おおよそ 1 分で公開 URL に反映され，手動でのアップロードは不要となる．これにより，配置作業由来の
ミスを防げる（\textbf{ヒューマンエラーの回避}）とともに，一度仕組みを作れば以降はサーバー配置を
意識せず内容の改善に専念できる（\textbf{開発への集中}）．

\section{今回構築したサービス}
\subsection{静的 Web サイト（Azure Static Web Apps）}
リポジトリ直下に \texttt{index.html} と \texttt{style.css} を置き，\texttt{.github/workflows/} に
デプロイ用 YAML を配置した．\texttt{main} への push を契機に GitHub Actions が実行され，成功時には
ログ末尾へ公開 URL が出力される（付録 図\ref{fig:deploy}）．実際に見出しの文字色を変更するなどの
修正を push しただけで本番サイトへ即座に反映された（付録 図\ref{fig:change}，図\ref{fig:static}）．
なお運用中，デプロイ用ワークフローが手動作成分と自動生成分の 2 本存在して衝突し，push のたびに一方が
失敗していたため，経路を 1 本に整理する重要性も学んだ．

\subsection{動的 Web アプリ（Azure App Service，加点課題）}
追加要件である「Web App Service（動的サイト）」の構築にも成功した．Node.js のフレームワークである
\textbf{Express} を用い，単なる静的ページの配信ではなく，\textbf{リクエストのたびにサーバー側で内容を
生成}するアプリを構築して Azure App Service（PaaS，無料 F プラン）へデプロイした．具体的には，
\textbf{現在のサーバー時刻の表示}，メモリ上で数える\textbf{アクセスカウンタ}，フォーム入力を
サーバー側で受け取って応答を返す\textbf{あいさつ機能}を実装した（付録 図\ref{fig:dynamic}）．
無料枠のため，初回アクセス時はコールドスタートで表示に時間がかかる（重い）仕様となっている．

\section{おわりに}
本課題を通じて，手動でのデプロイと自動デプロイの双方を理解した．手作業でのデプロイはミスが
起こりやすいが，GitHub Actions による CI/CD を用いることで push するだけで安全かつ短時間に公開できる．
さらに，Azure App Service を用いれば，サーバー管理を意識せずに動的な Web アプリまで公開できることを，
実際の構築を通して確認できた．

\section{付録}
\subsection{URL}
今回構築したサービスの公開 URL は以下の通りである（いずれも 2026 年 7 月 12 日 時点で稼働確認済み）．

静的サイト：https://orange-stone-0cfe86d00.7.azurestaticapps.net\\
動的アプリ：https://numata-25g1104-express-1833.azurewebsites.net
\end{quote}
※ 静的サイトのドメインには \texttt{.7.} が含まれる．これを省いた URL は 404 になるため注意．
