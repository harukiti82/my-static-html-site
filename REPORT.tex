\section{CI/CD とは}

\textbf{CI/CD} は「継続的インテグレーション（Continuous Integration）／継続的デリバリー・デプロイ
（Continuous Delivery / Deployment）」の略で、ソースコードの変更を\textbf{自動的にビルド・テスト・デプロイ}する
仕組みを指す。

\begin{center}
\begin{tabular}{lll}
\toprule
用語 & 意味 & 本レポートでの該当 \\
\midrule
CI & 変更を頻繁に統合し自動で検証する & \verb|git push| を契機に GitHub Actions が起動 \\
CD & 検証を通った成果物を自動で本番へ反映 & Azure Static Web Apps へ自動アップロード \\
\bottomrule
\end{tabular}
\end{center}

従来は「手元でファイルを作る $\rightarrow$ 手動で FTP アップロード」という流れだったが、
CI/CD ではこれを自動化する。本サービスでは \textbf{\texttt{main} ブランチへ push した瞬間に、
数十秒$\sim$1分で世界中に公開される}。

\section{システム構成}

静的な HTML / CSS のみで構成された Web サイトを、GitHub と Azure を連携させて自動公開している。

\begin{verbatim}
   [開発者 (ローカル)]
          |
          |  git push origin main
          v
   [GitHub リポジトリ (main ブランチ)]
          |
          |  push を検知してワークフローが起動
          v

          v
   [Azure Static Web Apps]
   グローバル配信 / CDN
          |
          |  HTTPS
          v
   [エンドユーザー]
\end{verbatim}

\section{CI/CD パイプラインの流れ}

\begin{enumerate}
  \item 開発者がローカルで \texttt{index.html} / \texttt{style.css} を編集する
  \item \verb|git commit| して \verb|git push origin main| する
  \item push を検知して\textbf{GitHub Actions のワークフローが自動起動}する
  \item \verb|Azure/static-web-apps-deploy| アクションがリポジトリ内のファイルを Azure へアップロードする
  \item Azure Static Web Apps がグローバル配信網に反映し、\textbf{約1分で公開URLに変更が反映}される
\end{enumerate}

プルリクエスト（PR）を作成した場合は、マージ前でも\textbf{プレビュー環境}が自動生成され、
本番に影響を与えずに変更内容を確認できる。PR をクローズするとプレビュー環境は自動で破棄される。

\section{デプロイ設定（ワークフロー定義）}

CI/CD の挙動は \texttt{.github/workflows/azure-static-web-apps-orange-stone-0cfe86d00.yml} に
定義されている（抜粋）。

\begin{verbatim}
name: Azure Static Web Apps CI/CD

on:
  push:
    branches:
      - main                       # main への push で本番デプロイ
  pull_request:
    types: [opened, synchronize, reopened, closed]
    branches:
      - main                       # PR でプレビュー環境を生成 / 破棄

jobs:
  build_and_deploy_job:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3          # (1) ソースを取得
      - name: Build And Deploy
        uses: Azure/static-web-apps-deploy@v1
        with:
          azure_static_web_apps_api_token: ${{ secrets.AZURE_..._TOKEN_... }}
          action: "upload"
          app_location: "/"                # (2) 公開するソースの場所
          output_location: "."             # (3) 成果物ディレクトリ
\end{verbatim}

\begin{itemize}
  \item \verb|on:| \dots\ いつ動くか（トリガー）。ここでは \texttt{main} への push と PR。
  \item \verb|secrets.AZURE_...TOKEN| \dots\ Azure が発行したデプロイ用トークン。GitHub の Secrets に
        安全に保管され、コードには一切含まれない。
  \item \verb|app_location| / \verb|output_location| \dots\ どのフォルダを公開するか。本サイトはビルド不要のため、
        リポジトリ直下（\texttt{/}）をそのまま配信する。
\end{itemize}

\section{考察 --- 運用で気づいた課題と改善案}

実際に運用してみたところ、\textbf{push のたびにワークフローが1回失敗している}ことが分かった
（第5章の履歴の「失敗」）。

\subsection*{原因}

リポジトリ内に\textbf{デプロイ用のワークフローファイルが2つ}存在している。

\begin{itemize}
  \item \texttt{azure-static-web-apps.yaml}（初期に手動作成したもの）
  \item \texttt{azure-static-web-apps-orange-stone-0cfe86d00.yml}（Azure が自動生成したもの）
\end{itemize}

両方が \texttt{main} への push で同時に起動し、\textbf{同じ Static Web App へ同時にデプロイしようとして衝突}する。
失敗した側のログには次のように記録されていた。

\begin{verbatim}
Status: InProgress. Time: 581.8(s)
Upload Timed Out. Unsure if deployment was successful or not.
\end{verbatim}

一方が先にデプロイを掴み、もう一方は待たされた末に\textbf{アップロードがタイムアウト}（約10分）して失敗している。

\subsection*{改善案}

不要になった \texttt{azure-static-web-apps.yaml} を\textbf{削除して1本化}すれば、
衝突がなくなり、毎回のデプロイが安定して成功するようになる。
CI/CD では「1つの成果物に対してデプロイ経路は1本」に保つことが重要だという学びが得られた。

\section{まとめ}

\begin{itemize}
  \item \textbf{CI/CD} により、\verb|git push| するだけでビルド$\sim$公開までが自動化され、手作業のアップロードが不要になった。
  \item \textbf{GitHub Actions + Azure Static Web Apps} の組み合わせで、静的サイトを無料かつ数十秒で世界公開できた。
  \item \textbf{PR プレビュー環境}により、本番に影響を与えず変更を確認できる仕組みも備わっている。
  \item 一方で、\textbf{ワークフローの重複}という運用上の落とし穴も体験し、「デプロイ経路の一本化」の重要性を学んだ。
\end{itemize}
